# Partner Insight Hub 项目目标说明

本文档用于说明 Partner Insight Hub 在 MVP / Hackathon 评审阶段需要证明的 6 个目标：

- 一个可演示 Demo
- 一个样例输入输出
- 一个最小业务流程
- AI 的具体处理步骤
- 当前未解决的问题
- 下一步是否值得继续推进的判断

这些目标共同回答一个问题：当前项目是否已经从「概念」走到「可验证的产品原型」，以及是否值得继续投入真实数据、部署和生产化工程。

---

## 1. 一个可演示 Demo

### 1.1 目标定义

可演示 Demo 指的是：评审人不需要阅读代码，也能通过浏览器看到完整产品入口、真实页面结构、样例数据、核心交互和 AI 助手能力。Demo 不要求接入生产数据，但必须能展示产品价值闭环。

### 1.2 当前实现

当前项目已经具备可演示 Demo：

- 前端入口：`/Partner_Insight_Hub/login`
- 登录后主界面：Dashboard App Shell
- 页面数量：16 个功能页面
- 数据来源：CSV、Excel、SQLite seed/mock 数据
- 后端服务：
  - FastAPI：登录、指标、转化、错误、联系人、渠道、AI
  - Node Express：订单、财务
- AI 助手：Dida API 页面右下角悬浮聊天组件

### 1.3 Demo 覆盖页面

| 分组 | 可演示页面 | 演示重点 |
|------|------------|----------|
| 业务指标 | 业务概览、业绩表现、业务报表 | 总体业务健康度、趋势图、月度分析报告 |
| 运营指标 | 技术指标、错误日志、转化指标 | 接口质量、错误定位、漏斗转化 |
| 渠道信息 | 渠道配置、渠道匹配、渠道热销、财务信息、订单日志、订单管理 | 渠道运营、订单查询、日志排障、账单风险 |
| Dida 营销 | 热销酒店推荐、直采推荐 | 渠道增值推荐能力 |
| 支持与帮助 | 联系方式、Dida API | 联系人维护、API 文档、AI 问答 |

### 1.4 Demo 启动方式

1. 启动 FastAPI：

```bash
cd backend
pip install -r requirements.txt
python init_db.py
python seed_integration.py
python seed_errors.py
uvicorn main:app --reload --port 8000
```

2. 启动 Node 订单/财务接口：

```bash
cd backend
npm install
npm run seed
npm start
```

3. 启动前端：

```bash
npm install
npm run dev
```

4. 访问：

```text
http://localhost:5173/Partner_Insight_Hub/
```

### 1.5 Demo 验收标准

- 可以打开登录页。
- 可以使用测试账号登录。
- 登录后可以进入 Dashboard。
- 左侧导航页面可以正常切换。
- 核心页面可以展示图表、表格或报告。
- 订单管理可以查询和导出。
- 订单日志可以查看详情和下载。
- Dida API 页面可以展示文档并打开 AI 助手。

## 2. 一个样例输入输出

### 2.1 目标定义

样例输入输出用于证明系统不是纯静态页面，而是具备明确的数据接口、参数结构、返回结构和错误处理方式。评审时可用它解释前后端如何协作。

### 2.2 登录样例

输入：

```http
POST /api/auth/login
Content-Type: application/json

{
  "email": "partner_a@example.com",
  "password": "Pass1234!"
}
```

成功输出：

```json
{
  "token": "<jwt-token>",
  "user": {
    "id": 1,
    "email": "partner_a@example.com",
    "channelName": "Booking Agency A",
    "contactName": "<contact-name>",
    "status": "active",
    "role": "user"
  }
}
```

失败输出：

```json
{
  "detail": "密码错误，请重新输入"
}
```

说明：

- 登录接口负责校验账号、密码和禁用状态。
- 前端根据返回 token 保存登录状态。
- 失败时展示明确中文错误提示。

### 2.3 订单查询样例

输入：

```http
GET /api/orders?page=1&pageSize=50&client_id=Agoda&refs=Agoda-123456
```

输出：

```json
{
  "data": [
    {
      "client_ref": "Agoda-123456",
      "dida_ref": "12345678",
      "channel_status": "Confirmed",
      "client_id": "Agoda",
      "dida_hotel_name": "Sample Hotel",
      "price": "$128.50"
    }
  ],
  "pagination": {
    "page": 1,
    "pageSize": 50,
    "total": 1,
    "totalPages": 1
  }
}
```

说明：

- `client_id` 用于渠道筛选。
- `refs` 支持批量订单号查询。
- 返回结果包含订单基础字段和分页信息。

### 2.4 AI 问答样例

输入：

```http
POST /api/chat/dida-api
Content-Type: application/json

{
  "messages": [
    {
      "role": "user",
      "content": "Booking Search 里哪些状态是最终状态？"
    }
  ]
}
```

输出示例：

```text
Status 2 Confirmed、Status 3 Canceled、Status 4 Failed 是最终状态。
Status 5 Pending 和 Status 6 OnRequest 不是最终状态，需要继续轮询 Booking Search。
```

说明：

- 前端以消息数组形式提交上下文。
- 后端基于 Dida API 内置知识提示词生成回答。
- 响应是流式文本，前端边接收边展示。

## 3. 一个最小业务流程

### 3.1 目标定义

最小业务流程不是覆盖所有功能，而是选出一条最能证明产品价值的主链路。该链路需要让评审人看到：渠道客户可以从登录开始，自助完成数据查看、问题定位和 API 支持。

### 3.2 推荐演示链路

```text
登录
  ↓
查看业务概览
  ↓
查看转化 / 错误 / 技术指标
  ↓
查询订单
  ↓
查看订单日志
  ↓
打开 Dida API 文档
  ↓
向 AI 助手提问
```

### 3.3 详细步骤

1. 渠道客户进入登录页。
2. 输入测试账号和密码。
3. 登录成功后进入 Dashboard。
4. 在业务概览查看核心指标和趋势。
5. 在转化指标查看查价、有价、验价、下单漏斗。
6. 在错误日志查看 Prebook / Book 错误。
7. 在订单管理输入订单号或批量订单号。
8. 查询订单结果并导出 CSV。
9. 在订单日志输入订单号。
10. 打开日志详情，查看 Price Confirm、Booking Confirm、Cancel 的 request/response。
11. 进入 Dida API 页面。
12. 向 AI 助手询问状态码、错误码或接入流程。

### 3.4 业务价值

- 对渠道客户：减少等待 OP 回复的时间。
- 对技术对接人：能自助查询 API 文档和错误含义。
- 对 OP：减少重复查询工作，把精力放在异常处理和策略优化上。
- 对 Dida：把数据、日志、财务和 API 支持集中成统一客户入口。

### 3.5 演示时长建议

| 阶段 | 时长 | 内容 |
|------|------|------|
| 入口介绍 | 30 秒 | 说明平台面向渠道客户 |
| 登录与看板 | 1 分钟 | 展示登录、业务概览、导航 |
| 数据查询 | 2 分钟 | 展示订单管理、订单日志、导出 |
| AI 助手 | 1 分钟 | 展示 Dida API 问答 |
| 总结价值 | 30 秒 | 回到问题和收益 |

## 4. AI 的具体处理步骤

### 4.1 目标定义

AI 目标不是做一个泛用聊天窗口，而是解决 Dida API 接入中的高频问题，包括鉴权、价格搜索、价格确认、创建预订、订单查询、取消订单、状态码和错误码。

### 4.2 用户侧体验

用户在 Dida API 页面右下角点击 AI 按钮，输入自然语言问题，例如：

```text
Booking Search 里哪些状态是最终状态？
```

前端显示 AI 的流式回答，用户不需要离开当前 API 文档页。

### 4.3 系统处理步骤

1. 用户在 `ApiChatBot` 输入问题。
2. 前端把当前消息和历史消息整理为 `messages`。
3. 前端调用：

```http
POST /api/chat/dida-api
```

4. FastAPI 接收请求。
5. 后端将用户问题与 Dida API system prompt 合并。
6. system prompt 内包含：
   - API v2.0 概览
   - 测试/生产 Base URL
   - ClientID + LicenseKey 鉴权方式
   - Price Search / Price Confirm / Booking Confirm / Booking Search / Cancel 流程
   - 订单状态码
   - 2xxx 定价错误码
   - 3xxx 预订错误码
   - Go-Live 流程和最佳实践
7. 后端通过 Claude CLI 子进程调用 `claude-sonnet-4-6`。
8. Claude 生成回答。
9. FastAPI 使用 `StreamingResponse` 流式返回。
10. 前端用 `ReadableStream` reader 逐块读取内容。
11. 前端实时更新助手气泡。
12. 请求失败时显示失败兜底提示。

### 4.4 处理边界

AI 助手适合回答：

- Dida API 接入流程。
- API 鉴权格式。
- 订单状态码含义。
- 错误码含义。
- Booking Search / Cancel 等接口注意事项。
- Go-Live 流程。

AI 助手不适合直接执行：

- 修改账号权限。
- 查询真实生产订单。
- 发起真实取消订单。
- 修改渠道配置。
- 处理财务付款。

### 4.5 后续增强方向

- 将 Dida API 文档从硬编码 prompt 改成可更新知识库。
- 增加引用来源，让回答能指向具体文档章节。
- 增加问题分类和拒答策略。
- 将 Claude CLI 替换为稳定服务端模型调用。
- 增加对话日志和质量评估。

## 5. 当前未解决的问题

### 5.1 问题清单

| 问题 | 当前影响 | 优先级 | 建议处理 |
|------|----------|--------|----------|
| 生产数据尚未接入 | Demo 只能证明流程，不能证明真实业务口径 | P0 | 选 2-3 个渠道做真实数据试点 |
| FastAPI 与 Node 后端并存 | 部署和维护复杂度较高 | P0 | 收敛服务边界，优先统一 API 网关 |
| 权限隔离仍是 MVP 粒度 | 生产环境存在数据越权风险 | P0 | 增加租户级数据隔离和权限校验 |
| AI 依赖 Claude CLI | 部署环境不稳定，难监控 | P1 | 封装为稳定模型服务 |
| GitHub Pages 只能部署前端 | 后端仍需独立环境 | P1 | 明确 ECS/容器部署方案 |
| 报表中心不够配置化 | 只能展示固定分析报告 | P2 | 设计报表模板、字段选择和调度能力 |
| 自动化测试不足 | 回归风险较高 | P2 | 补登录、查询、导出、AI smoke test |

### 5.2 风险说明

当前风险主要集中在生产化工程，而不是产品价值。换句话说，MVP 已经能说明「客户自助门户」这件事有价值，但还不能直接作为生产系统上线。

最关键的上线前风险是：

- 数据口径是否和真实业务一致。
- 每个渠道是否只能看到自己的数据。
- 后端部署是否稳定。
- AI 回答是否可控、可追踪、可更新。

## 6. 下一步是否值得继续推进的判断

### 6.1 结论

值得继续推进。

### 6.2 判断依据

| 判断维度 | 当前状态 | 结论 |
|----------|----------|------|
| 用户痛点 | 指标查询、订单排障、API 支持都是真实高频需求 | 成立 |
| Demo 完整度 | 已覆盖登录、数据、订单、日志、财务、API、AI | 成立 |
| 技术可行性 | 前后端和样例接口已经跑通 | 成立 |
| AI 适配度 | Dida API 问答场景边界清晰，适合 AI 助手 | 成立 |
| 生产风险 | 数据、权限、部署、AI 服务仍需增强 | 可控但需投入 |

### 6.3 推荐推进策略

下一阶段不要继续横向扩页面，而应纵向做扎实：

1. 真实数据试点：选择 Agoda 或 2-3 个渠道接入真实指标、订单、日志和财务数据。
2. 后端收敛：统一 FastAPI / Node 的职责，减少部署复杂度。
3. 权限隔离：明确用户、渠道、Client ID、数据范围的访问控制。
4. AI 稳定化：把 Claude CLI 封装为可监控的服务，增加知识库更新机制。
5. 部署验收：确定前端、后端、数据库、环境变量和日志监控方案。
6. 自动化测试：覆盖登录、核心页面、订单查询、日志详情和 AI 问答。

### 6.4 阶段性里程碑

| 阶段 | 目标 | 验收结果 |
|------|------|----------|
| MVP Demo | 证明完整演示链路 | 已完成 |
| Pilot | 接入少量真实渠道数据 | 待推进 |
| Beta | 完成权限隔离、后端部署、AI 服务化 | 待推进 |
| Production | 面向真实渠道开放 | 待评估 |
